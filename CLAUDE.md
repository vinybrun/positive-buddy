# habit_buddy — quick reference

Flutter (Riverpod + Drift) habit companion. Android-only device work.

## Device / deploy
- Device id: `ROGI4LBAOV8HXOFA` (Poco X7 Pro, HyperOS / POCO launcher `com.mi.android.globallauncher`). Two adb transports show up — always pass `-s ROGI4LBAOV8HXOFA`.
- Build+install: `flutter build apk --debug && adb -s ROGI4LBAOV8HXOFA install -r build/app/outputs/flutter-apk/app-debug.apk` (install in place, never uninstall — HyperOS shows an approval dialog).
- **Free RAM after building.** `flutter build apk` leaves a Gradle daemon (~600 MB) + Kotlin compile daemon (~160 MB) running in the background — they use `android-studio/jbr/java` but are NOT the IDE. When done with device/build work for the session, stop them: `cd android && ./gradlew --stop`, then kill any stragglers (`pkill -f GradleDaemon; pkill -f KotlinCompileDaemon`). Don't leave them idling between tasks.
- Screenshots are full-res 1220×2712; `input`/crop coords are in those pixels. `uiautomator dump /sdcard/u.xml` → buddy tiles carry `content-desc` = label (Russ/Sage/Skye/Mira/Bento/Lumi/More).
- Debug page: long-press the Today app-bar title (long-clickable node ~`[52,130][700,312]`, center `376 221`; hold ≥800ms). Has "Create test habit", "Pin Small/Big/Tiny", "Fire test notification" (uses LIVE settings), log helpers.
- App is **portrait-locked** (`screenOrientation=portrait` in manifest + `setPreferredOrientations` in main.dart).
- Launcher icon swap disables the `MainActivity` component, so `am start .../.MainActivity` fails. **Always launch via** `adb -s … shell monkey -p dev.viniciusbrun.habit_buddy -c android.intent.category.LAUNCHER 1`. Check the live icon with `cmd package resolve-activity --brief -c android.intent.category.LAUNCHER <pkg>`.

## Build toolchain gotcha
- AGP 9.0.1 + `home_widget` 0.9.2: the plugin skips applying KGP under AGP 9, so its Kotlin doesn't compile → "Unresolved reference es.antonborri…". Fix lives in `android/build.gradle.kts`: a `subprojects` block re-applies `org.jetbrains.kotlin.android` to `:home_widget` and pins its Kotlin jvmTarget to 1.8. Remove when the plugin ships an AGP-9 release.

## Home-screen widgets (tiny/small/big)
- Pure logic + payload: `lib/features/widgets/widget_data.dart` (color math, keys `wb_*`, `WidgetSnapshot`). Tested in `test/widget_data_test.dart` + `test/widget_snapshot_test.dart`.
- Sync + interactivity: `lib/features/widgets/home_widget_service.dart` — `buildSnapshot(db)` reads state, `render(db)` pushes via `home_widget` + updates 3 providers; reactive stream listeners started from `main.dart`. Big-widget Done buttons fire `widgetInteractivityCallback` (logs `manual_done`, source `widget`).
- Native: `android/app/src/main/kotlin/.../{WidgetRender,SmallHabitWidgetProvider,BigHabitWidgetProvider,TinyHabitWidgetProvider}.kt`. `WidgetRender` rasterizes buddy circle + gradient bar + ring at the widget's real px size. Buddy loaded from `flutter_assets/<assetPath>`.
- Layouts `res/layout/widget_*.xml`, info `res/xml/widget_*_info.xml`, manifest receivers declared tiny→small→big. Picker previews are baked PNGs in `res/drawable-nodpi/preview_{tiny,small,big}.png` (real on-device crops).
- Prefs: `widgetColorMode` (default **primary**; alt `progressive` = red→yellow→green) and `widgetShowCount` (hide "done/total" + center bar). Toggles in You → "Buddy & look". Big widget bg follows app `composeBackground` (light/dark/tint).
- **Big-widget card bg = a pre-colored rounded BITMAP** (`big_bg` ImageView, `WidgetRender.roundedCard`), NOT a tinted white shape. A runtime-tinted white shape flashes WHITE for one frame on re-inflation (white paints before the tint lands); a bitmap's only intermediate is transparent. `widget_big.xml` root (FrameLayout) carries NO static bg — don't add one.
- **Flash-on-Done-tap, the real story**: each `updateAppWidget` makes the HyperOS launcher clear the whole widget surface to wallpaper for ~130ms (a blink). This is UNAVOIDABLE from the app (proved: full vs partial update, bitmap vs solid bg, multi-view vs single-ImageView all blink — even the small widget's opaque buddy blinks; it's just invisible on the transparent small/tiny tiles). So the only lever is **update COUNT**. A Done tap fires the BG isolate's render AND the app's stream listeners re-fire the same state ~250ms+ later → 2+ updates → 2+ blinks ("second flash ~1s later"). **Native per-provider dedupe (static `lastSig`) was tried and REVERTED — do NOT re-add it.** Skipping `updateAppWidget` when the sig is unchanged makes the widget go PERMANENTLY BLANK/INVISIBLE after a Done tap: the launcher has already cleared the surface expecting a redraw, so a no-op leaves it cleared. Every provider must ALWAYS call `updateAppWidget`. The second blink is suppressed at the Dart source (NOT by skipping the native redraw): the BG Done handler pings `notificationRefreshPortName`; main.dart's port listener calls `HomeWidgetService.notifyExternalRender()`, which sets a 1500ms TTL flag; the next stream-debounced render inside that window is dropped (flag consumed, so later real changes still render). The BG isolate's render always stands. (Cross-isolate `saveWidgetData`/`getWidgetData` is also unreliable: the BG isolate's writes are NOT read back by the main isolate in time, so a Dart-side prefs dedupe does NOT work either.)
- **Tap-to-open**: tiny + small open the app on tap (whole `widget_root`); big opens ONLY from the header (`big_header` = buddy + bar) — the rows below own their Done taps, the row body has no launch handler. All use `WidgetRender.launchIntent(context)` = `getLaunchIntentForPackage` → resolves the **currently-enabled launcher alias**. Do NOT use `HomeWidgetLaunchIntent.getActivity(.., MainActivity::class)`: the stage-icon swap disables `MainActivity`, so an explicit intent to it resolves to nothing and the tap does nothing.

## Buddy picker (You page) — `lib/features/profile/buddy_picker_widget.dart`
- Compact mode shows first 3 of `settings.effectiveBuddyOrder` + "More"; rest hide behind More. Order persists in `user_profile.buddyOrderJson` (v15, JSON id array; empty = default = staged species first). Parse/encode helpers on `UserSettings`.
- **Reorder-on-settle**: picking a buddy hidden behind More floats it to front, but only on a *settle* event — More closed, page popped (`dispose`), or app paused (`WidgetsBindingObserver`). Never while the tray is open. Writes via a captured `_repo` so dispose() is safe.
- **Evolution ring**: `_EvolutionRingPainter` draws a ring in the buddy's OWN seed color (not theme primary) filling toward the next stage. Fraction = `BuddyScoringEngine.progressToNextStage(totalScore)`. Only for `hasStages` species (fox/snake/bird).

## Stage-aware launcher icon (level-up swap)
- Evolving species get one alias per stage: `MainActivity<Species>S1..S5` (fox/snake/bird); static species keep one alias (cat/dog/butterfly). All in manifest between `STAGED-ALIASES-START/END`. Component selection logic in `MainActivity.kt` (`STAGED_SPECIES`/`STATIC_ALIASES`/`ALL_ALIASES`).
- `LauncherIconBridge.queueForBuddy(buddy, stage:)` queues; applied on `AppLifecycleState.paused` (swap mid-use would kill the task). Queued on buddy change (`profile_repository` reads maxStage) and on level-up (`BuddyProgressRepository.reconcile`, gated on stage increase; reconcile is only ever called for the active buddy).
- Stage icon foregrounds are generated from `assets/buddies/<sp>/stages/stage_N.png` into `mipmap-*/ic_launcher_<sp>_s<n>_fg.png` (Python/PIL, ~66% safe-zone) + `mipmap-anydpi-v26/ic_launcher_<sp>_s<n>.xml` (bg `@color/buddy_<sp>_bg`). Stage art is placeholder/80px — regen when real art lands.

## HyperOS widget quirks (verification)
- `requestPinAppWidget` returns true but the POCO launcher silently does NOT place the widget. To place: long-press the app icon **on the home screen** → "Add widgets" → swipe to the size → "Add to Home screen" (or drag the preview card onto home).
- Launcher caches widget label/description across reinstall; **reboot** to refresh (`adb reboot`, then `wm dismiss-keyguard` + swipe up). previewImage refreshes without reboot.
- The home long-press "Widgets" tray (`com.mi.globalminusscreen`) only lists App-Vault widgets, not standard AppWidgetProviders — don't look there.

## DB cleanup (no sqlite3 on device; local one at `~/Android/Sdk/platform-tools/sqlite3`)
- force-stop → `run-as <pkg> cat app_flutter/habit_buddy.sqlite` to /tmp → edit with `PRAGMA wal_checkpoint(TRUNCATE); …` → `adb push` to `/data/local/tmp` → `run-as cp` into place → `run-as rm` the `-wal`/`-shm`. Drift stores DateTime as unix seconds; opaque ARGB ints arrive in Kotlin prefs as `Long`.

## Per-buddy notification sounds (custom sounds)
- One notification channel per buddy (channel sound is immutable post-create). Logic: `resolveReminderChannel(UserSettings)` in `lib/features/notifications/local_notification_service.dart` → channel id `habit_reminders_<species>_<buddySoundVersion>`, raw sound `res/raw/buddy_<species>.ogg`. Channels created lazily (`_ensureBuddyChannel`) only for the active buddy.
- Active only when `soundEnabled && customSoundsEnabled && selectedBuddy != null`; else default channel (system sound gated by `soundEnabled`). Setting `customSoundsEnabled` = DB v16, default **true**, toggle in You→Behavior under "Sound".
- **Sounds are PLACEHOLDER tones** (distinct ffmpeg-synth beeps per species, so on-device routing is audible). Real candidates in repo-root `animal_sounds_library/<animal>/` (+ `MANIFEST.md` license/URLs). Priority buddies = cat/dog/fox/snake/bird/butterfly(silent — see its README).
- **To install a chosen sound**: crop+convert with `ffmpeg -i <src> -ss <in> -to <out> -ar 44100 -ac 1 -c:a libvorbis -q:a 4 android/app/src/main/res/raw/buddy_<species>.ogg`, then **bump `buddySoundVersion`** in local_notification_service.dart (else installed devices keep the stale channel sound). Test stays green via `test/buddy_sound_channel_test.dart`.
- Verify on device: `adb -s … shell dumpsys notification --noredact | grep -A1 habit_reminders_<sp>` → `mSound=android.resource://…/raw/buddy_<sp>`; `mLastAudiblyAlertedMs` set = it played.
